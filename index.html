How the Web Request Process Works

When a user opens a website, the browser does not magically know where everything is. It follows a chain of steps that includes DNS lookup, HTTP messaging, server processing, and browser rendering.

Step-by-step request cycle

These actions usually happen in seconds.

  1. The user enters a URL. The browser reads the address and decides what resource is being requested.
  2. DNS resolves the domain. The browser finds the IP address for the domain name.
  3. A connection is opened. The browser and server establish a network path. If the site uses HTTPS, a TLS handshake is also part of this stage.
  4. The browser sends an HTTP request. This request asks for a page, image, stylesheet, script, or another file.
  5. The web server processes the request. It decides whether to serve a static file, run code, call an application, or return an error.
  6. The server sends an HTTP response. The response includes a status code, headers, and often a response body.
  7. The browser renders the page. It parses HTML and then loads linked CSS, images, fonts, and scripts.

What comes before HTTP?

HTTP is central, but it is not the only thing happening.

  • DNS: maps a readable domain like example.com to the server's IP address.
  • TCP or QUIC: provides the transport layer underneath the HTTP conversation.
  • TLS: encrypts data when a site uses HTTPS.

This is why page speed depends on more than just the HTML file itself. Lookup time, connection setup, encryption, and the number of resources all matter.

What comes after the first response?

One page usually leads to many more requests.

The first HTML file often tells the browser to fetch additional assets such as stylesheets, JavaScript files, images, fonts, and video. A page that looks like a single thing to the user can actually be assembled from many individual requests and responses.

Browsers can also cache files. If an asset is already cached and still fresh, the browser may not need to download it again.

What is inside an HTTP request?

A request tells the server what the client wants.

  • Method: the action being requested, such as GET or POST
  • Target resource: the path or URL for the file or endpoint
  • Headers: details about accepted formats, language, caching rules, or the browser itself
  • Body: optional data, often used with forms or APIs

What is inside an HTTP response?

A response tells the browser what happened and sends data back.

  • Status code: a short result such as 200 or 404
  • Headers: metadata about the response, caching, type, and security rules
  • Body: the actual content such as HTML, JSON, an image, or a file download
Common status codes
Status code Meaning Example
200 Success The page or file was found and returned correctly.
301 Moved permanently The browser is redirected to a new address.
404 Not found The resource does not exist at the requested path.
500 Internal server error The server encountered a problem while processing the request.

What affects web speed?

Performance is shaped by both network and server behavior.

Latency

The longer it takes signals to travel between browser and server, the slower requests feel.

Number of files

More images, scripts, and fonts usually mean more work for the browser and the network.

Caching

Good caching reduces repeated downloads and makes repeat visits much faster.

Compression

Compressed files move faster because the browser has less data to download.

Why developers should care

Understanding the request cycle helps developers debug broken pages, missing files, slow loading times, and server-side problems. It also makes later topics such as APIs, databases, deployment, and performance optimization much easier to understand.
