The Evolution of HTTP

HTTP started as a very simple protocol for fetching documents. As sites became larger, faster, and more interactive, the protocol had to improve too. Each major version solved a different kind of communication problem.

Version highlights

These short summaries show why each version mattered.

HTTP/0.9

The earliest version was extremely basic. It only supported the GET method and had no status codes or headers.

HTTP/1.0

This version added status codes and headers, which made responses clearer. It still opened a new connection for each request.

HTTP/1.1

This update introduced persistent connections and better caching behavior. It stayed dominant for many years because it solved a huge performance problem.

HTTP/2

HTTP/2 introduced multiplexing and binary framing so multiple transfers could share one connection more efficiently.

HTTP/3

HTTP/3 maps HTTP semantics onto QUIC. That change lowers latency and helps more on weak or unstable networks.

HTTP version comparison
Version Main improvement Transport note Main reason to move forward
HTTP/0.9 Simple document retrieval Very limited request and response format No headers, no status codes, and only one method
HTTP/1.0 Headers and status codes New connection per request Too much overhead for pages with many resources
HTTP/1.1 Persistent connections and improved caching Still runs over TCP Modern pages pushed it toward more efficient multiplexing
HTTP/2 Multiplexing and binary framing Still relies on TCP underneath Needed better behavior under packet loss and lower setup latency
HTTP/3 Uses QUIC with lower latency Runs over UDP through QUIC Improves performance and connection recovery on modern networks

What stayed the same

New versions changed delivery, not the core idea.

  • Browsers still send requests and servers still send responses.
  • Status codes, methods, and headers still matter.
  • The basic goal is still resource delivery between client and server.

What changed the most

Efficiency became the main theme of HTTP history.

  • Fewer wasted connections
  • Better handling of many files at once
  • Lower latency on real-world networks
  • More practical performance for mobile browsing

Why newer versions matter

Protocol changes directly affect user experience.

Faster first load

Lower latency and better connection handling help pages start rendering sooner.

Better support for complex sites

Modern pages pull many assets, and newer HTTP versions are better at handling that complexity.

Stronger modern networking

HTTP/3 works with QUIC, which improves connection behavior on unstable networks and wireless connections.

Big idea

HTTP history is really the story of the web adapting to its own success. As users expected richer pages, stronger security, and faster performance, the protocol had to keep getting smarter without breaking the basic request-and-response model that made the web work in the first place.
